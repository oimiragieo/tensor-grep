# API and Data Contracts

This document defines the backward-compatibility guarantees for data structures and CLI outputs used by enterprise integrations, IDEs, and editor-plane agents.

release_docs_current_tag: v1.84.0

## 1. Configuration (`sgconfig.yml`)
The root-level keys and structure of `sgconfig.yml` are guaranteed to be stable within a major version. Unrecognized keys will be ignored rather than causing fatal errors to allow progressive rollout of new configurations.

## 2. AST Cache (`project_data_v6.json`)
The schema of the `.tg_cache/ast/project_data_v6.json` cache is versioned within its filename.
- Backward compatibility is NOT guaranteed across major/minor versions.
- If the schema changes (for example to `v7`), `tensor-grep` will automatically invalidate older cache files and rebuild them transparently.

## 3. Text-search compatibility
The stable text-search compatibility contract is the validated compatibility set covered by the parity suite and contract benchmark runner. `tg search --help` may expose more rg-style flags, but only the rows below and explicitly named deterministic edges are part of the public rg-compatibility claim.

Current validated rows:
- `-i/--ignore-case`
- `-v/--invert-match`
- `-C/--context`
- `-A/--after-context`
- `-B/--before-context`
- `-g/--glob`
- `-l/--files-with-matches`
- `--files-without-match`
- tensor-grep aggregate `--json`
- ripgrep JSON Lines via explicit `--format rg --json`
- `--ndjson`
- `-F/--fixed-strings`
- `-w/--word-regexp`
- `-m/--max-count`
- `--files`
- `-t/--type`
- `-./--hidden`
- `-L/--follow`
- `-S/--smart-case`
- `-n/--line-number`
- `--column`
- `-c/--count`
- `--count-matches`
- `-a/--text`
- `-0/--null`
- `-U/--multiline`
- `--multiline-dotall`

Character-for-character identity is not required for help formatting, but command presence, supported rows, accepted normalization, and the deterministic parity corpus are part of the public contract. Additional rg-style flags may be exposed in `tg search --help`, but they are not covered by the stable compatibility claim until they are added to the contract suite and benchmark runner.

Agent automation contracts:
- `tg search PATTERN` defaults to the current directory when no path is provided.
- Invalid regex syntax exits as an error distinct from no-match and emits a diagnostic that recommends `--fixed-strings` for literal searches.
- `tg search --json` emits a valid tensor-grep aggregate JSON object even when there are zero matches.
- `tg search --format rg --json` forwards to ripgrep and emits rg JSON Lines events without a tensor-grep envelope for tools that require rg's event schema.
- `tg search --files-with-matches` stays root-based on the ripgrep path instead of expanding large candidate-file lists into the Windows process argument vector, and plain path-list output emits one trailing line separator only.
- `tg search` with `--format rg` is a public exact ripgrep-style text formatter for automation that needs rg-shaped stdout. Pair it with `--sort path` when deterministic path ordering matters across backends.
- The native front door treats `--format rg` as a no-op for rg-compatible text output and preserves `--sort path` when forwarding to ripgrep. Non-rg `--format` search output remains a Python CLI formatting surface.
- The public native front door must not reject flags advertised by the public Python CLI. It may execute the request natively, forward to ripgrep, or route to the Python sidecar, but sidecar routing must be intentional and covered by a public-native regression test.
- Root shortcut invocations with option-first common search flags, including `tg -t js PATTERN PATH` and `tg --count-matches PATTERN PATH`, are treated as `tg search ...` and must preserve those flags through native, ripgrep, or sidecar routing instead of falling back to positional-only parsing.
- Cached-session requests must stay on the warm path by default. One-shot `tg session ...` commands and daemon-routed session requests validate files captured in the session snapshot with direct size/mtime checks and must not walk the full repository merely to check for added files. Added-file discovery belongs to explicit refresh paths: `tg session refresh ...` and requests that opt into `--refresh-on-stale`. Daemon response-cache hits intentionally use `detect_added_files=False` before serving cached `context-render` / `edit-plan` payloads so truncated implicit sessions stay fast; new files under the repo are invisible to response-cache invalidation until `refresh_on_stale` or `tg session refresh`. Daemon `stats` exposes `response_cache_stale_detection = "snapshot_mtime_only"` and `response_cache_added_file_detection = false` so agents can see the tradeoff without guessing. The 5 symbol commands (`defs` / `impact` / `refs` / `callers` / `blast-radius`) are also response-cached (audit #113): a repeated warm request with an identical symbol/provider/max-tests/max-depth/max-repo-files is served straight from the daemon's response cache instead of re-running the caller-scan or blast-radius graph walk. Unlike `context-render` / `edit-plan`, the symbol-command gate honors the request's real `refresh_on_stale` flag -- added-file detection included -- before trusting a cache hit, because a caller/blast-radius answer that silently omits a newly-added call site is a wrong answer, not merely a stale one; a genuinely new file (or a modified/removed tracked file) busts a symbol-command cache entry on the very next request. Session edit-plan graph ranking, test matching, and blast-radius metadata must stay bounded to selected context budgets and report the bounded radius scope in `edit_plan_seed.blast_radius_scope`; daemon requests should keep a short connect probe while allowing longer edit-plan/context responses. Top-level native-provider `tg context-render` and `tg edit-plan` may reuse an already-running daemon and create an implicit session keyed by root and `max_repo_files` so repeated agent-loop calls hit daemon response-cache counters without reusing a smaller scan budget for later larger requests. `tg session list` and `tg session daemon status` should discover nearby session scopes when the requested/current root has no direct session metadata, and JSON output should expose whether discovery was used. Daemon-routed relative paths are resolved against the daemon root before session lookup so `tg session edit-plan ID . --daemon` works from the served project root and does not accidentally target the daemon process working directory.
- `tg checkpoint list PATH` must keep default discovery nearby and bounded: direct scope, ancestors, and immediate non-generated children only when the direct scope is empty. Recursive child-scope discovery is opt-in through `--discover`; generated/cache roots require the explicit `--discover-full` mode. Explicit recursive discovery may cache fingerprinted checkpoint index paths for repeat calls, but `create` and `undo` must invalidate ancestor discovery caches so agent loops do not reuse stale scope lists after checkpoint state changes.
- `tg search --files` remains a generated-root guardrail surface. The native front door may route it to the Python sidecar so broad generated-root refusal, `--glob`, `--type`, `--max-depth`, and `--allow-broad-generated-scan` keep the same public semantics.
- `tg search --multiline` / `-U`, `--multiline-dotall`, and `-0/--null` must be accepted by the native front door and forwarded to the backend that preserves rg-compatible behavior for the requested output mode.
- `--sort path` is the deterministic contract for `--files-with-matches`, `--files-without-match`, `--replace`, and path-list automation. The parity suite also checks match, no-match, parse-error, and binary-skip exit-code behavior.
- Default output ordering for root-scale `--files-with-matches`, `--count`, and `--force-cpu` is a semantic result parity contract, not a golden stdout ordering contract. Use `--sort path` when automation needs deterministic path ordering across backends.
- Broad generated roots such as `.claude` are routed through Python guardrails instead of rust-first/native passthrough so generated `.claude/context` snapshots can be pruned by default.
- Unbounded broad generated-root scan requests are refused with exit code `2` when hidden file-list scans contain generated, cache, or dependency child directories, or when content searches directly target a generated/cache/dependency root. Explicit `--no-ignore` content searches over an ordinary project root follow ripgrep and may traverse ignored generated child directories; direct generated roots still require a scoped path, `--glob`, `--type`, `--max-depth`, or `--allow-broad-generated-scan`. Unbounded searches against a parent that looks like a multi-project workspace root are also refused before the broad walk. `tg scan` / MCP ruleset scans apply the same safety posture to AST/ruleset workflows: direct temp, AppData, user-home, filesystem-root, generated/cache/dependency, or multi-project workspace roots must be scoped to a project/subdirectory, bounded with `--max-depth` (and optionally `--glob` / `--type`), or explicitly opted in with `--allow-broad-generated-scan` before any scanner walk begins.
- An implicit-path `tg search` (`PATH` omitted, defaulting to the current directory per the earlier bullet) whose directory WALK exceeds an internal file-count ceiling (1500 files visited, counted independent of match count or glob/type selectivity) is refused with exit code `2` and an actionable message instead of running unbounded. This guard runs as the first statement of `execute_ripgrep_search` itself, so it applies uniformly across every entry point that can reach that function -- the full Python CLI, the standalone native binary's default search front door (including its `-e`/`--regexp` form), the positional CLI, `tg-search-fast`, and the PyO3 FFI bridge -- rather than one call site that a different entry point could bypass. An explicit, deliberately-scoped `PATH` is never refused regardless of size.
- `tg search --type-list` prints a built-in fallback list when no ripgrep or standalone native binary is available. `--pcre2-version` follows ripgrep semantics and exits with an error when no PCRE2-capable backend is available.
- `tg ast-info --json` exposes AST language identifiers via `{"languages": [...]}` for agent discovery without help-text scraping.
- Read-only `tg run` follows search-style no-match semantics: zero AST matches exit `1` while successful matches exit `0`; parse/configuration errors remain distinct non-zero failures. On Windows, if the received pattern is still wrapped in single quotes, `tg run` warns that `cmd.exe` treats those quotes literally and suggests double quotes for `cmd.exe`.
- On Windows, PowerShell automation should invoke `tg` or `tg.ps1` for regex metacharacters. PowerShell double quotes expand `$NAME`-style values before tensor-grep receives argv, so literal patterns containing `$` should use single quotes or escape `$` as `` `$ ``. Direct `.cmd` invocation from PowerShell is not a safe argv-preserving contract for unescaped metacharacters such as `|` because `cmd.exe` parses the line before the batch wrapper receives arguments; `tg.cmd` is for `cmd.exe`.
- The quoted multi-word no-match pattern case from `cmd.exe`, direct `tg.cmd`, and Python `subprocess.run([...])` is a public launcher contract. The launcher must preserve the phrase as one argv item and must not turn it into a shorter false-positive search plus bogus paths.
- Stable managed install scripts should prefer the matching release-native CPU `tg` binary as the public front door when the GitHub release asset is available, set `TG_SIDECAR_PYTHON` for Python-backed commands, set `TG_NATIVE_TG_BINARY` for the sidecar, and fall back to `python -m tensor_grep` when no native asset exists. On Windows, the managed native front-door directory must be placed on User PATH ahead of compatibility shim directories so `cmd`, unprofiled PowerShell, and `subprocess.run(["tg", ...])` resolve `~/.tensor-grep/bin/tg.exe` before the slower argv-safe `tg.cmd` bridge. If a tensor-grep-owned `tg.com` bridge is copied into another PATH directory to outrank a foreign same-directory `tg.exe`, the bridge must still find `~/.tensor-grep/.venv` for sidecar-backed commands and must point `TG_NATIVE_TG_BINARY` back at the managed front door rather than the bridge copy.
- `tg doctor --json` must make launcher routing observable. It should report `path_tg_first_launcher_kind` for the current process PATH, `fresh_shell_path_tg_first_launcher_kind` for Windows fresh-shell PATH when available, `python_subprocess_path_tg_first_launcher_kind` for Python `subprocess.run(["tg", ...])` style resolution, and `path_tg_launcher_warning` when the current process still resolves a compatibility shim even though a fresh shell would resolve the managed native front door. It also exposes `shell_escaping_guidance` so automation can surface PowerShell `$` expansion and `cmd.exe` metacharacter escaping guidance without scraping README prose. This warning is diagnostic: it means restart or refresh PATH before timing subprocess-heavy workflows. Python subprocess resolution is its own contract because Windows `CreateProcess` can choose a foreign same-directory `tg.exe` even when shells prefer a tensor-grep `tg.com` bridge through `PATHEXT`.
- `tg doctor --json` must also make GPU search routing observable. `gpu.search_runtime_probe` should run a small fixed-string `tg search --gpu-device-ids ... --json` probe through the resolved native front door and report `requested_gpu_device_ids`, `routing_backend`, `routing_reason`, `sidecar_used`, `routing_gpu_device_ids`, and a `status` that is `supported` only for `NativeGpuBackend` with `sidecar_used = false`.
- If the first PATH or Python-subprocess `tg` reports a version string that is not tensor-grep-shaped (`tg ...` or `tensor-grep ...`), `tg doctor --json` should classify it as `foreign`, set `path_tg_first_is_foreign`, `fresh_shell_path_tg_first_is_foreign`, or `python_subprocess_path_tg_first_is_foreign`, and emit a warning plus remediation field such as `path_tg_foreign_warning`, `path_tg_foreign_remediation`, `fresh_shell_path_tg_foreign_warning`, `fresh_shell_path_tg_foreign_remediation`, `python_subprocess_path_tg_foreign_warning`, and `python_subprocess_path_tg_foreign_remediation`. Foreign launchers are not tensor-grep-owned stale launchers: readiness should fail with explicit remediation, and installer/doctor code must not delete or overwrite unrelated tools. The explicit repair path is `tg repair-launcher`: it removes verified tensor-grep-owned Python Scripts entrypoints and backs up self-identifying orphaned tensor-grep Python Scripts entrypoints that shadow the managed native front door. With `--allow-foreign-rename`, it is Windows-only, backs up the first foreign Python-subprocess `tg.exe` to a `.bak` file, installs the verified managed native front door into that same PATH slot, and must refuse to act on a foreign launcher without the allow flag.
- Stable installers must build replacement managed environments and front-door files in a staging directory and only swap them into `~/.tensor-grep` after package installation and front-door generation succeed. PowerShell installer native commands must check `$LASTEXITCODE` before the staged swap. A failed package resolve/install must preserve the previous managed install and shims.
- `tg upgrade` must not infer "latest PyPI version" solely from unchanged local metadata. It should refresh package metadata, skip yanked PyPI releases when selecting an exact version, pin the latest same-or-newer PyPI version when known, verify the target Python can import `tensor_grep`, and report a verification error instead of success if the sidecar is corrupted. The scheduled Windows self-upgrade helper must run the same import/version verification before writing a success log. Managed installs must also refresh the managed release-native front door to the verified sidecar package version, refresh stale tensor-grep-owned `tg.com` PATH bridges when present, or schedule a Windows native-front-door retry helper when the running native `tg.exe` is locked.
- `tg classify --format json` is a public sidecar command even when invoked through the native front door. Default `classify` should use deterministic local heuristics without probing tokenizer/model providers. JSON output includes additive top-level `classification_backend` provenance with `provider_requested`, `provider_used`, `provider_status`, `fallback_reason`, and `cache.status` / `cache.offline` / `cache.local_files_only` so agents can distinguish local, provider, cache, offline, and fallback results without scraping stderr. JSON classification rows include `label`, `confidence`, `file`, `path`, 1-based `line`, and `snippet` so agents can map labels back to source evidence. Operators must opt into the CyBERT/Triton provider with `TENSOR_GREP_CLASSIFY_PROVIDER=cybert`; provider failures still fall back before tokenization or Hugging Face model loading so agent calls fail fast and quietly. MCP `tg_classify_logs` follows the same local-first provider contract.
- GPU benchmark correctness compares result sets, not only successful match exits. `rg` exit code `1` is a valid no-match outcome, so no-match patterns must be accepted as correctness passes when `tg` also returns no matches. The GPU scale gates should include 1GB and 5GB rows and exact match/file-set correctness for every >=1GB GPU corpus before any GPU promotion claim. GPU auto-recommendation must remain false unless required 1GB/5GB correctness checks pass and a selected GPU beats both `rg` and `tg_cpu` at the required scale and declared workload class. Explicit `--gpu-device-ids` routing must not initialize or warn about unselected GPUs, unsupported-device inventory warnings must not be attached to unrelated selected-GPU timing rows, and scale benchmarks must record the runtime backend actually handling `--gpu-device-ids`. Sidecar-routed GPU requests are not valid native CUDA scale-gate timings, and the artifact contract must include `fallback_or_sidecar_counts_as_gpu_proof = false`.
- Public managed GPU promotion additionally requires a managed NVIDIA native front door with release provenance. Installers and `tg upgrade` write `tg-native-metadata.json`; `benchmarks/run_gpu_native_benchmarks.py --public-managed-proof` must require `native_frontdoor_metadata_status = present`, `native_frontdoor_metadata_version` matching the expected public `tg` version, a nonempty NVIDIA `native_frontdoor_asset_name`, `NativeGpuBackend`, `sidecar_used = false`, native 1GB/5GB direct-`rg` route/correctness, and an advanced many fixed-string proof gate that beats the fair single-invocation `rg -F -e ... -e ...` baseline. Promotion artifacts must expose `public_managed_promotion_ready` and `public_gpu_proof`; local CUDA-feature builds and single-pattern speed wins are implementation evidence, not public managed proof.
- GPU benchmark artifacts must make proof status machine-readable. Top-level `gpu_evidence_status`, `gpu_proof`, `native_gpu_unavailable`, and `not_gpu_proof_reason` distinguish promotion-ready native CUDA evidence from CPU fallback or sidecar compatibility output. `scale_gate_summary.workload_taxonomy` and `scale_gate_summary.promotion_evidence_contract.promotion_scope = "declared_workload_class_only"` record that many fixed-pattern claims require a fair single-invocation `rg -F -e ... -e ...` baseline and that resident repeated-query claims remain candidate/not-measured until a benchmark exists. Per-row `promotion_evidence = false` and `not_gpu_proof_reason` are required for unsupported GPU rows.
- Cold-path benchmark artifacts must distinguish the configured launcher experiment from the command kind actually being timed. `run_benchmarks.py` records `environment.tg_launcher_mode` and `environment.tg_launcher_command_kind` so native-exe, `.cmd` shim, `uv`, and Python-module timings are not mixed into one speed claim. It must also record `tg_binary_version_status`; a stale in-tree native tg binary is a claim blocker by default in benchmark scripts unless the operator explicitly passes `--allow-claim-unsafe-launcher` for exploratory timing. Benchmark artifacts should emit top-level warnings when timings include shim or interpreter overhead.
- Agent workflow benchmark artifacts must keep capsule/edit-loop evidence separate from raw search-speed evidence. `run_agent_workflow_benchmarks.py` records the literal positioning `agent-native workflow benchmark; not a cold exact-text speed claim`, then reports `agent_capsule` contract metrics and `edit_loop` search/plan/apply/verify timings. Use it to evaluate confidence, alternatives, validation alignment, snippets, rollback, and edit order, not to imply `tg` beats `rg` for cold exact-text search.
- Agent success harness artifacts must keep end-to-end agent success evidence separate from raw search-speed evidence. `run_agent_success_harness.py` writes `artifacts/bench_agent_success_harness.json`, records the literal positioning `agent-native end-to-end success harness; not a raw search speed claim`, reports `workflow_surfaces` for `intent`, `context`, `edit_seed`, `apply`, `verify`, and `rollback`, and refuses stale in-tree native `tg` binaries by default unless `--allow-claim-unsafe-launcher` marks the run as exploratory. Use it to evaluate whether a task can move from query intent to checkpointed edit and rollback, not to imply `tg` beats `rg` for cold exact-text search.
- The fast agent-readiness dogfood gate is `python scripts/agent_readiness.py --output artifacts/agent_readiness.json`; `tg dogfood --output artifacts/dogfood_readiness.json` wraps the same gate with a verdict envelope for agent and CI logs and writes the nested readiness child report beside the explicit output as `artifacts/dogfood_readiness.agent-readiness.json`. For the current `v1.84.0` release line it checks public shell version probes, sidecar-backed public launcher probes, `public-windows-launcher-quoted-patterns`, the installed-public `public-search-advertised-flag-sweep` for advertised rg-style search flags and option-first root forwarding, repo doctor sanity, launcher route diagnostics including foreign launcher ownership, `context_consistency`, deterministic rg parity edges, broad generated-root scan guardrails, AST smoke, MCP context-render smoke, the `agent-capsule` CLI/MCP smoke, the `agent-capsule-mixed-language` trust regression, the `agent-capsule-hardcases` polyglot monorepo/noisy generated-root regression, docs claim hygiene, the managed native-upgrade contract, and the public positioning that `rg` remains the cold exact-text baseline while `ast-grep` remains the structural-search feature/performance baseline. The dogfood report also emits `world_class_readiness.status = "not_claimed"` with explicit limitation surfaces such as `raw_cold_text_search`, `full_ast_grep_surface`, `public_gpu_acceleration`, `lsp_semantic_provider`, and `agent_target_selection_metrics`; a PASS verdict is fast release-readiness evidence, not proof of full rg replacement, full ast-grep replacement, production GPU acceleration, production LSP navigation, or enterprise target-selection accuracy.
- AST workflow compatibility is scoped. `tg run` is a validated AST slice for structural search and guarded rewrites; read-only `tg run` accepts and routes the accepted ast-grep semantic option slice (`--selector`, `--strictness`, `--stdin`, and repeated `--globs`) through the wrapper path with honest routing. JavaScript/TypeScript standalone class-method patterns may be contextualized through a synthetic class wrapper so users can search for `static method(...) { ... }` without writing the parent class shape. `--files-with-matches` is a read-only text output mode and must reject JSON or rewrite/apply combinations. `tg scan`, `tg test`, and `tg new` are tensor-grep workflow surfaces for bounded rule/config operations, not an ast-grep replacement. Help text and docs must describe unsupported ast-grep flags as roadmap gaps instead of implying full CLI parity.

Context and edit-planning contracts:
- `context-render` and MCP context output must not let `edit_plan_seed.primary_file`, `navigation_pack.primary_target.file`, rendered source sections, and follow-up reads contradict each other.
- `context_consistency` reports whether the primary file is included, whether rendered context matches the primary target, whether confidence was downgraded, and why a primary file was omitted from rendered budget.
- The default JSON/LLM render profile includes executable body lines for selected functions. Compact rendering can remove comments, docstrings, blank lines, type-only imports, and boilerplate, but it must preserve matched behavior unless the caller requests a summary-only profile in a future contract.
- Validation command hints use `validation_plan[].detection` with values of `detected`, `heuristic`, or `generic`. JavaScript package-manager commands require `package.json` evidence; Python commands require tests, project markers, or Python layout evidence; omit commands entirely when no runner evidence exists instead of inventing `npm test` or `uv run pytest`.
- Validation hints must align with the selected primary target language unless verified cross-language dependency evidence exists. `validation_alignment` records whether validation remained aligned or whether incompatible commands were filtered; a TypeScript primary target must not silently receive pytest-only validation, and a Python primary target must not silently receive JS-only validation.
- Edit validation command templates support `$file` and `{file}` placeholders. For applied rewrites, placeholders are replaced with each edited file path and the command runs once per edited file; commands without a file placeholder run once against the original target working directory. Examples should quote the placeholder, such as `python -m py_compile "$file"`, so Windows paths with spaces remain one argv item.
- `edit-plan` and `context-render` JSON both expose top-level `validation_commands` copied from `navigation_pack` or `edit_plan_seed` for quick agent access. Agents should not need command-specific parsing just to find the recommended validation command list. `edit-plan` JSON additionally exposes a top-level `validation_plan` (the structured step list -- `command`, `confidence`, `detection`, `runner`, `scope`, optional `target` -- copied from `edit_plan_seed.validation_plan`), matching the shape `agent` JSON already surfaces at its own top level; this is additive and never removes or mutates `validation_commands`.
- `edit-plan` JSON also exposes top-level `confidence` (`{overall, downgrade_reasons}`) and `ask_user_before_editing` (`{required, reasons}`), matching the shape `agent` JSON already surfaces at its own top level. Both are derived through the SAME trust-check + confidence + ambiguity ladder `agent` uses (shared helpers in `agent_capsule.py`, called from `repo_map._edit_plan_confidence_and_ask`) -- not a separately invented heuristic -- so the >=0.75 no-ask threshold, the query-language and validation-alignment downgrades, the scan-truncation ask-gate, AND the alternative-target-tie confidence cap (0.74) + unrequested-marker-helper ask-reason all read identically across both commands. `ask_user_before_editing.required` is the hard auto-edit safety gate, so it MUST match `agent` on ambiguity: the tie and marker-helper signals depend only on the ranked-alternatives (`candidate_edit_targets`/`file_matches`) and the query -- not on rendered snippets -- so `edit-plan` computes them, and validation-based tie RESOLUTION is applied too (it reads `edit-plan`'s own `validation_plan`). Only the genuinely snippet-/call-site-/LSP-evidence-gated enrichments stay `agent`-only, because `edit-plan` renders no source text and has no such evidence to corroborate against: the graph-corroborated + token-budget confidence UPLIFTS, the LSP confidence BOOST and LSP-based tie RESOLUTION, and the snippet-only / rendered-context-consistency ask-reasons (every ambiguity cause `edit-plan` can have already contributes its own direct ask-reason, so this never under-reports). This is an honest omission, not a regression.
- Repo-map and context-ranking defaults exclude generated/cache/dependency directories, binary files, logs, and hidden non-code files from normal code context.
- Future token-efficiency profiles must be opt-in and recoverable. Do not mutate raw `--format rg`, `--json`, or `--ndjson` contracts to save tokens; add explicit agent profiles with hard budgets, grouped excerpts, omission counts, and refetch commands.
- Compact output should cap breadth before cutting semantic payload. Selected functions must preserve matched lines and executable body slices, and omissions should be reported in metadata or clearly non-code delimiters instead of comment-shaped placeholders inside source blocks.
- `tg agent` / Actionable Context Capsule output is a separate agent contract, not a changed search contract. `capsule_version = 1` payloads include primary file/function, alternative targets for plausible non-primary candidates, route rationale, bounded source snippets with line maps, validation evidence with provenance, risk/edit order, checkpoint or rollback metadata, omission counts, confidence, follow-up read commands, call-site evidence status, optional `gpu_acceleration` route evidence, and an "ask user before editing" recommendation when uncertainty or risk is high. Capsule v1 leaves `related_call_sites` empty unless verified call-site evidence is explicitly collected. Future capsule changes should be additive. When the underlying repo scan was truncated, the capsule additively carries the same scan-side fields the other code-intelligence commands expose — `scan_limit`, `scan_remediation`, `partial`, `partial_reason`, `deadline_limit`, and `result_incomplete` — only when present; a pure render/token output cap on the capsule's own snippet budget never sets these. `partial_reason` is currently always `"deadline"` when present on `agent`/`context`/`context-render`/`edit-plan` output (mirrors `tg codemap`'s own `partial_reason` field, which additionally distinguishes `"scan_limit"`/`"self_verify"` for its own coverage contract): `partial = true` on these four commands is a strictly time-budget signal, so a `partial_reason = "deadline"` payload can be substantively complete and correct — every stage that ran found everything it looked for — but still reports `partial` because SOME stage, checkpointed or not, pushed total elapsed time past the caller's `--deadline` before the payload returned. This is a deliberate false-partial: each command's builder rechecks the shared absolute deadline one final time at its own return point (regardless of which internal stage actually consumed the time) and would rather flag a technically-complete-but-late response as `partial` than silently report `exit 0` on a response that blew its declared time budget. `partial_reason = "deadline"` is therefore NOT the same signal as `result_incomplete` (which means data was actually dropped by a `--max-repo-files` / count cap) — a payload can be `partial` with `partial_reason = "deadline"` and NOT `result_incomplete`, or vice versa, or both at once.
- When `tg agent`'s `deadline_limit` is present (per the false-partial rationale above), it additively carries `assembly_stages_skipped` — a list naming which POST-collection ASSEMBLY stage(s) (currently `"vendored_subtree_detection"`, `"suggested_scope"`, `"context_pack_assembly"`) were skipped because the shared `--deadline` budget was already exhausted by the time that stage would have started. This is distinct from `scan_limit`/`deadline_limit.files_scanned`, which describe the COLLECTION (repo-map walk/parse) stage only: `assembly_stages_skipped` is present only when at least one assembly stage actually cut work short, so a capsule with no assembly-tail impact stays byte-identical to before this field existed, and it is always nested under `deadline_limit` (never a standalone top-level key implying honesty independent of `deadline_exceeded`).
- Agent GPU evidence is opt-in through explicit selected device IDs. It may batch query terms through `tg search --gpu-device-ids ... -e ...` to collect extra evidence, but it must report `NativeGpuBackend` with `sidecar_used = false` before the capsule treats that evidence as used. Sidecar-routed GPU output is compatibility evidence only and must be reported as unsupported, not as GPU acceleration or a speed claim.
- Search JSON and NDJSON should distinguish requested GPU IDs from routed GPU IDs. `requested_gpu_device_ids` records the user-selected IDs; `routing_gpu_device_ids` records the devices actually used by the runtime route. A request that falls back to `NativeCpuBackend` with `gpu-auto-fallback-cpu` must emit `routing_gpu_device_ids = []`; that is CPU fallback, not GPU acceleration proof. JSON envelopes for explicit GPU requests should add `gpu_evidence_status`, `gpu_proof`, `native_gpu_unavailable`, and `not_gpu_proof_reason` so harnesses do not infer GPU proof from the presence of `--gpu-device-ids`.
- Capsule confidence must be honest when query language hints, exact symbol intent, primary target language, selected snippets, and validation commands disagree. In mismatch cases, both `confidence.overall` and `primary_target.confidence` should be capped, `ask_user_before_editing.required` should become true, and `context_consistency` should expose `query_language_hints`, `primary_target_language`, `validation_alignment`, and `validation_filtered_count`. Unresolved equal-confidence alternatives should expose `alternative_confidence_tie`, `alternative_confidence_tie_count`, and `tied_alternative_targets`, cap confidence below the normal edit threshold, require confirmation, and mirror the decision in top-level `ambiguity.status = "tie_requires_confirmation"`; validation-resolved ties must record `ambiguity.status = "tie_resolved"`, `resolved_by = "targeted-validation"`, and concrete `resolution_evidence` commands, while LSP-resolved ties require explicit provider-response proof. A `tie_requires_confirmation` capsule additively narrows `suggested_scope` when it is not already populated by a scan-limit truncation: it prefers the same centrality-weighted directory rollup `tg orient` exposes, and falls back to the tied candidates' own deepest common parent directory when that whole-repo signal is flat/tied; `suggested_scope` stays `null` (never a guess, never the scan root) when neither source finds a strict subdirectory to narrow to.
- A token-budget-only rendering cut with a graph-corroborated primary must not be reported as resolution uncertainty. When the ONLY reason capsule confidence dropped is a render/token-budget artifact (`payload.truncated`, or the capsule's own snippet budget cutting a non-primary source) and blast-radius call-site collection independently corroborates the primary (the primary matches the query and verified caller evidence was collected), `confidence.overall` and `primary_target.confidence` may rise to `0.8` with `context_consistency.confidence_basis = "resolution-quality"`, and the budget-only `ask_user_before_editing` reasons are cleared so `required` becomes `false`. This uplift must never fire for a genuine misroute (the primary was never ranked/selected, or does not match the query), a genuine equal-confidence alternative-target tie, a language-mismatch trust cap, or an unrequested marker-helper demotion — any of those must still cap confidence and keep `ask_user_before_editing.required = true`.
- Capsule and search-intent routing evidence should label claims as `parser-backed`, `rg-backed`, `graph-derived`, `heuristic`, `LSP-confirmed`, or `stale/uncertain`. If these signals disagree, confidence should be downgraded and the contradiction should be explicit instead of hidden behind a single ranked file.
- LSP provider availability is not semantic proof. `tg doctor --with-lsp` provider entries must distinguish install availability from request health with `health_status` and `health_check`. Provider-backed navigation payloads must include `lsp_proof`, `lsp_evidence_status`, and `not_lsp_proof_reason` when `lsp` / `hybrid` mode falls back to native evidence or returns no provider evidence. A row only counts as `lsp_proof` when it carries an explicit `lsp_provider_response = true` marker from a completed provider request; `provenance = "lsp-*"` alone is not proof. Use `tg lsp --debug-trace <language> --path <workspace>` to capture a one-shot JSON-RPC health trace, timeout budgets, status, and stderr tail when provider startup or semantic probes fail.
- `context-render`, `edit-plan`, and `agent` keep semantic providers opt-in through `--provider native|lsp|hybrid`. Default `native` behavior must not start LSP providers. When opted in, the payload records `semantic_provider`; primary targets may carry `lsp_provider_response`, `lsp_proof`, `lsp_operation`, and `lsp_resolution_basis` only when an actual provider request confirms the native planning anchor.
- `TG_CAPSULE_LSP_CONFIDENCE_BOOST=1` temporarily opts `tg agent` into hybrid primary-target proof for capsule confidence only. When the selected primary target has row-level LSP proof on an allowlisted provider language, equal-confidence alternatives can be marked `ambiguity.status = "tie_resolved"` with `resolved_by = "lsp"` and confidence remains capped at `0.85`; lower trust caps still win.
- Agent command surfaces must accept both explicit flags and positional aliases for the user-facing target. Query commands (`context`, `context-render`, `agent`, `edit-plan`, and matching `session` commands) accept `PATH QUERY` or `QUERY`; symbol commands (`defs`, `source`, `impact`, `refs`, `callers`, `blast-radius*`, and matching `session` commands) accept `PATH SYMBOL` or `SYMBOL` while keeping `--query` / `--symbol` as stable automation flags. Supplying both forms for the same target is invalid.
- Symbol-command exit codes are a three-state agent contract: **`0`** = complete result (trust it); **`1`** = genuine not-found on a *complete* scan (the symbol/result truly is absent); **`2`** = INCOMPLETE — the scan was truncated by `--deadline` (`partial: true` + `partial_reason: "deadline"` + `deadline_limit`) or a `--max-repo-files` cap (`result_incomplete: true`), so a zero or small count is NOT the full answer; parse the JSON and retry with a larger budget/cap or a narrower scope. `tg agent`, `tg context`, `tg context-render`, and `tg edit-plan` each recheck the shared absolute deadline one final time at their own return point, so `partial`/`partial_reason` reflect ANY stage overrunning the budget — not only the specific sibling stages each command happens to instrument — and can fire even when every stage that ran completed and found everything it looked for (see the capsule contract above for the full false-partial rationale). An agent must never treat an exit-2 result (or its possibly-empty list) as complete. This mirrors `tg search`'s `2 = result_incomplete` convention. The same three-state contract covers `tg map`, `tg context`, `tg context-render`, `tg edit-plan`, `tg blast-radius-render`, and `tg agent` — **including their warm-daemon fast-paths** (`tg session daemon start` + `--daemon`/auto-routed requests, task #108 extended `agent`'s fast-path to match): a scan-truncated `scan_limit.possibly_truncated` / `caller_scan_limit.possibly_truncated` / `partial` / `caller_scan_truncated` payload still prints its full JSON or rendered text (never swallowed), then exits `2`, on the daemon route exactly as on the cold route. `tg orient` (task #108 also gave it a warm-daemon fast-path) has NO exit-2 contract at all, cold or warm — it never gates on `_scan_incomplete`; a truncated `tg orient` scan still exits `0`, surfacing `scan_limit`/`truncated`/`suggested_scope` as informational fields only, not a retry signal. `tg orient --deadline` (CLI consistency fix, added for command-surface parity so an agent that learned `--deadline` on the symbol commands does not get a Click "No such option" exit-2 here) is the same story: a deadline-truncated scan additively surfaces `partial`/`deadline_limit` (mirroring `_copy_partial_signal`'s shape) purely as informational fields, never flipping orient's exit code. An OUTPUT-only cap (`output_limit.possibly_truncated` from `--max-files` on `map`, or `--max-callers`/`--max-files` render caps) is a COMPLETE analysis capped only for display and stays exit `0`; only a SCAN truncation exits `2`.
- `--deadline` (CLI consistency fix, CEO v1.72.1 dogfood M1) is additionally accepted on the true symbol commands `tg source` and `tg blast-radius-plan` — previously Click-rejected as an unknown option despite already sharing the same `_emit_symbol_command_result` (`source`) / `_scan_incomplete` (`blast-radius-plan`) three-state exit-code contract described above as `defs`/`impact`/`refs`/`callers`/`blast-radius`; both thread `deadline_seconds` into `build_symbol_source`/`build_symbol_blast_radius_plan` the same additive way (`build_repo_map`'s existing `deadline_monotonic` parameter), so a truncated scan now correctly surfaces `partial`/`deadline_limit` and exits `2` on both. It is also accepted on `tg docs-coverage` (both the default coverage report and `--stale` mode): a `--deadline`-caused walk truncation is a NEW, separate signal from the command's pre-existing `--max-repo-files` count-cap (`scan_limit.possibly_truncated`, unchanged, still exits `0`) — only the new `partial`/`deadline_limit` time-budget signal exits `2`, checked *before* `--check`'s own exit-`1` gate so truncation trumps a (possibly wrong) confident coverage/stale count, mirroring the found-vs-truncated precedent above. `tg imports` also accepts `--deadline` but is a documented NO-OP: it resolves a single file's own imports (O(1), no repo scan to bound), so the flag exists solely so an agent that learned `--deadline` works elsewhere does not get a Click "No such option" exit-2 here; the value is never threaded anywhere and output is byte-identical with or without it.
- Task #108 (Tier-2 daemon moat): `tg orient` and `tg agent` join the warm-session-daemon fast path (task #94's `TG_SESSION_DAEMON_AUTOSTART`, default ON; no separate flag) via `orient`/`agent` daemon commands, `build_orient_capsule_from_map`/`build_agent_capsule_from_map`, and dedicated response-cache key builders (`_orient_response_cache_key`/`_agent_response_cache_key`) that key on every output-affecting flag, mirroring the 5 symbol commands' anti-bleed discipline (audit #113). Both are added-file-sensitive: an implicit session honors the caller's `refresh_on_stale` for added-file detection, not the context-render/edit-plan `detect_added_files=False` shortcut, because a new central file (orient) or a new call site (agent) changes the answer, not just its staleness. Two correctness traps, both audit-#107-class (a daemon-served map-based lookup disagreeing with what a fresh cold scan would find): **TRAP A** — `agent`'s call-site-evidence collector runs `build_symbol_blast_radius_from_map` against the daemon's cached map, which has no literal-seed rescue (unlike the cold `build_symbol_blast_radius` wrapper, which retries via `_literal_symbol_seed_files` on a truncated no_match); when that condition fires, the response carries an internal `daemon_evidence_unreliable: true` sentinel and the client (`main._maybe_agent_via_running_daemon`) discards the whole response and falls back to cold instead of serving call-site evidence it cannot trust. **TRAP B** — `suggested_scope` is computed in the `build_context_render` WRAPPER (not `build_context_render_from_map`); `build_agent_capsule_from_map` replicates that exact block against its own map so a warm capsule does not silently drop `suggested_scope` on a truncated scan. `--gpu-device-ids` on `tg agent` never routes to the daemon (the GPU evidence probe shells out to a fresh `tg search` subprocess, which must not run inside a daemon worker thread) — it always takes the cold path.
- `--deadline` on the cold path of `tg agent`, `tg context-render`, `tg edit-plan`, `tg orient`, and `tg codemap` (closes #197/#200's front-door residual) is measured from **CLI command entry**, not just the underlying repo scan: `main.py`'s `_cli_deadline_monotonic` anchors the absolute `time.monotonic()` deadline at the TOP of the command body — before the lazy builder import, path/query resolution, GPU-id parsing, and the warm-daemon-skip gate — so that front-door time is budgeted the same way scan time already is. The five cold-path builders (`build_agent_capsule`, `build_context_render`, `build_context_edit_plan`, `build_orient_capsule`, `build_codemap`) accept an optional pre-anchored `deadline_monotonic` override that takes precedence over recomputing a fresh one from `deadline_seconds`; direct callers that only pass `deadline_seconds` (the MCP tool, the session daemon's own `_from_map` variants, tests) are unaffected — the fallback computation is byte-identical to the prior behavior. `tg agent`'s own separate cold-path-only default (`DEFAULT_AGENT_CLI_DEADLINE_SECONDS`, "dogfood finding 1 (F4)", applied only when neither `--deadline` nor `--no-deadline` is passed) is unaffected by this fix and keeps its existing anchor point, computed after the daemon-miss decision — that generous default bound is not the small-explicit-`--deadline` case this fix targets. The remaining process-startup prefix — interpreter boot plus Typer/Click argument dispatch, which runs before Python even reaches the command body — is a separate, irreducible ~100-200ms gap that no CLI command body can account for; it is not included in any `--deadline` budget.
- `tg route-test` (#223) runs the FULL `context-render` AND `edit-plan` builds back to back to compare their primary targets, so — unlike `context-render`/`edit-plan` themselves, which stay unbounded by default — it is the one command in this family with a cold-only DEFAULT `--deadline` bound: it reuses `DEFAULT_AGENT_CLI_DEADLINE_SECONDS` (`tg agent`'s existing 60s cold-path constant) rather than inventing a second default, because the composite cost was observed overrunning an external harness's own timeout under concurrent load even though a single side alone finished well inside it. Both builder calls share ONE pre-anchored `deadline_monotonic` (anchored at CLI command entry, before path/query resolution) — side 2 (`edit-plan`) naturally receives whatever wall-clock side 1 (`context-render`) left rather than each side getting its own fresh budget. `--no-deadline` disables the bound entirely; an explicit `--deadline` overrides the default. Honesty: when EITHER side's builder payload stamps its own `partial` (a deadline truncation), route-test's composite JSON additively stamps top-level `partial: true`, `partial_reason: "deadline"`, `deadline_limit` (`{deadline_exceeded, context_render, edit_plan}` naming which side(s) truncated), and `agreement_basis: "partial"` — the tell that the reported `agreement` boolean was computed from one or two truncated sides and must not be trusted as a full-confidence verdict just because the two sides happen to still agree — then exits `2` through the same `_scan_incomplete` gate the sibling commands use. `route-test` has no warm-daemon fast path (cold-only), and a complete comparison (neither side truncated) omits all four additive fields, staying byte-identical to the pre-#223 payload.

Known current limitations:
- Explicitly opted-in broad generated-root walks can still be expensive. The Python path-list output path and managed Windows launchers force UTF-8, but scope file-list commands to the smallest useful root whenever possible.
- Broad generated roots remain agent-hostile when callers opt in to them. Use scoped paths, globs, file types, and `--max-depth` for `tg search` before reaching for opt-in; `--max-repo-files`, `--max-callers`, and `--max-files` are code-intelligence command budgets, not `tg search` flags.
- `impact --symbol` is a broader planning signal and can be noisier than `blast-radius`; use `blast-radius` for direct symbol impact checks.

## 8. EvidenceReceipt signing (`tg evidence emit --sign` / `tg evidence verify`)
`tg evidence emit` always attaches a top-level `receipt_sha256` -- a keyless SHA-256 integrity/dedup/chain digest, present whether or not the receipt is signed. Passing `--sign` additionally attaches `signing` and `signature` blocks; passing `--previous <path>` attaches `previous_receipt_sha256` independent of `--sign`.
- **Algorithm**: Ed25519 (`cryptography`'s `Ed25519PrivateKey`/`Ed25519PublicKey`), chosen because a downstream verifier (e.g. gotcontext) is a separate trust domain from the machine that emits receipts -- an asymmetric signature lets it verify without ever holding a key that could forge one. `tg audit`'s HMAC-SHA256 (`audit_manifest.py`) remains correct for its own same-operator local tamper-evidence use case.
- **Canonicalization (`tg-canonical-json-v1`)**: `json.dumps(receipt_without_signature_and_receipt_sha256, sort_keys=True, separators=(",", ":"), ensure_ascii=True, allow_nan=False).encode("utf-8")`. Only the `signature` and `receipt_sha256` fields are excluded from the signed/hashed bytes -- the `signing` block itself (including its `algorithm` claim) is signed, so an algorithm-downgrade or public-key-swap attack invalidates both the digest and the signature. `receipt_sha256` and the Ed25519 signature are computed over the identical canonical bytes.
- **Number formatting is CPython-specific (cross-implementation caveat)**: `tg-canonical-json-v1` inherits CPython's `json` number formatting -- notably its `float.__repr__` (shortest round-trip, e.g. `0.1 + 0.2` serializes as `0.30000000000000004`) and its arbitrary-precision integer output. This is *not* RFC 8785 / JCS canonical-number formatting. A verifier written in another runtime (gotcontext / JavaScript / Go) MUST reproduce CPython's exact number serialization to recompute the same canonical bytes, or its digest/signature check will spuriously fail on any receipt containing a non-trivial float. Practical guidance: `tg`'s own receipt fields are simple (small integers, short decimals like a `confidence` of `0.72`, and strings), which round-trip identically across implementations; a downstream verifier should either match CPython number formatting exactly or treat numeric fields with care (and any caller-supplied `--cost-json` should avoid high-precision floats). A future `tg-canonical-json-v2` could adopt JCS if a cross-language verifier needs stricter number canonicalization.
- **Wire shape when signed**: `"signing": {"algorithm": "ed25519", "key_id": "sha256:<hex of the raw 32-byte public key>", "public_key": "<base64 raw 32 bytes>", "signed_at": "<RFC3339>", "canonicalization": "tg-canonical-json-v1", "receipt_id": "<uuid4>"}` plus `"signature": {"value": "<base64 signature>"}`. The signature block holds only the raw signature value; the signer's identity lives in the (signed) `signing` block.
- **Trust bootstrap (never trust an embedded key for authenticity)**: `tg evidence verify` always reports the signer's key fingerprint (`key_id`), computed from the embedded public key's own bytes -- never from a claimed label. An embedded public key only proves internal self-consistency (anyone can generate a keypair and sign with it); actual authenticity requires an out-of-band pinned key, supplied via repeatable `--trusted-key <base64-public-key>` or `TG_EVIDENCE_TRUSTED_KEYS` (comma-separated). `--require-trusted` fails `valid` closed unless the embedded key's fingerprint matches a pinned entry (`hmac.compare_digest`); a trusted-key check against an unsigned receipt is always invalid, mirroring `audit_manifest.py`'s "signing key supplied but manifest unsigned" rule.
- **Key management**: the private key defaults to `~/.tensor-grep/keys/evidence_ed25519.key` (per-user home, never the per-repo `.tensor-grep/` that `tg audit` writes to), overridable via `--signing-key` / `TG_EVIDENCE_SIGNING_KEY`. `tg evidence keygen` writes the private key at file mode 0600 via an atomic create-then-rename (refusing to overwrite an existing file or write through a symlink unless `--force`) and the public key alongside at `<path>.pub` (0644). `tg evidence pubkey` prints the public key and `key_id` for registration with a downstream verifier.
- **Fail-closed**: `--sign` with no resolvable signing key (or an unavailable `cryptography` install) exits non-zero and writes no receipt -- it never falls back to emitting an unsigned receipt. Every receipt-file read is size-bounded (rejects a file over a few megabytes) before parsing/hashing: both the primary receipt (`tg evidence verify <path>`) and the `--previous` chain file (reachable from both `tg evidence verify --previous` and `tg evidence emit --previous`).
- **Consumer note**: signatures do not expire; replay/staleness detection (via `receipt_id`, `signed_at`, and the `revision.commit_sha`/`dirty_tree_sha256` repo binding) is the downstream verifier's responsibility, not `tg`'s.

## 9. Ledger claims (`tg ledger claim` / `release` / `list`) -- advisory, code-scoped agent coordination (Slice 1)

`tg ledger` is a local, ADVISORY, code-scoped agent-to-agent coordination plane. A claim is a lightweight, TTL-expiring, best-effort record that one agent intends to touch a symbol or file -- it is **never a lock on an edit**. `tg ledger claim` always succeeds (exit `0`) even when a live overlapping claim exists from another agent; overlap information is reported for the caller to act on, and `tg` never refuses or blocks an edit because of it. State lives at `<root>/.tensor-grep/ledger/claims/index.json` and is created only by the first `claim` in a repo (default-inert: `list`/`release` never create it).

**Status: EXPERIMENTAL / preview (Slice 1)**, unlike section 8's EvidenceReceipt above -- the CLI surface and JSON schema documented in this section may change in a minor release. `tg ledger --help` and each subcommand's own `--help` are marked `EXPERIMENTAL (Slice 1)` for the same reason, mirroring how `tg find`'s docstring is prefixed `EXPERIMENTAL:`.

- **Exit-code contract (two states only, not the three-state symbol-command contract in section 3 above)**: `0` = the command completed and its JSON/text output can be trusted, including a claim with non-empty `overlaps`, a release that matched zero claims, and a list with zero live claims -- none of those are errors. `2` = a fail-closed condition fired and **nothing was written**: a claims-index lock-acquire timeout, a symlink at the index destination, a `--files` entry that resolves outside the repo root (`../...`, an absolute path, or a symlink escape), a corrupt or oversized on-disk index, or missing required arguments (`claim` with neither `--symbol` nor `--files`; `release` with neither `--claim-id` nor `--symbol`). There is no exit `1` for this command family.
- **Claim record shape**: `{ledger_schema_version, kind: "claim", claim_id, agent_id, symbols, files, intent, note, created_at, expires_at, ttl_seconds, revision}`. `claim_id` is `claim-<UTC-compact-timestamp>-<8-hex>`. `revision` is the same `{status, commit_sha, branch, dirty, dirty_tree_sha256, dirty_file_count}` / `{status: "unavailable", reason}` shape section 8's EvidenceReceipt uses (`_repo_revision_identity`) -- captured for free at claim time so a caller can tell whether an overlapping claim was filed against the same commit.
- **`tg ledger claim PATH [--symbol S ...] [--files a,b] [--intent edit] [--ttl SECONDS] [--agent-id ID] [--note TEXT] [--json]`**: `--symbol` is repeatable; `--files` is one comma-separated value of root-relative paths (traversal-refused, per the exit-2 contract above). Requires at least one `--symbol` or `--files` entry. Response JSON: `{version, schema_version, routing_backend: "Ledger", routing_reason: "ledger-claim", sidecar_used: false, ledger_schema_version, advisory: true, claim: {...}, overlaps: [...]}`. `overlaps` lists LIVE (non-expired) claims from OTHER agent_ids whose `symbols`/`files` literally intersect the new claim -- never the caller's own other claims. Each overlap entry is `{claim_id, agent_id, symbols, files, intent, expires_at, revision_matches}`; `revision_matches` is `true`/`false` only when both sides have a resolved git identity, and `null` (never fabricated) when either side's revision capture came back `unavailable`.
- **`tg ledger release PATH (--claim-id ID | --symbol S) [--agent-id ID] [--json]`**: `--claim-id` releases an exact claim regardless of which agent owns it (the opaque id itself is the authorization -- e.g. an orchestrator garbage-collecting a crashed sub-agent's claim). `--symbol` releases only the resolved `--agent-id`'s OWN live claim(s) covering that symbol, so a common/guessable symbol name can never release another agent's claim by accident. Response JSON: `{..., routing_reason: "ledger-release", released: [...], released_count}`. Releasing zero matching claims (already released, already expired, or never existed) is not an error.
- **`tg ledger list PATH [--symbol S] [--agent-id ID] [--json]`**: pure read -- prunes expired entries for display only and never writes (so `list` alone can never create `.tensor-grep/ledger/`). Filters combine as AND. Response JSON: `{..., routing_reason: "ledger-list", claims: [...], count}`.
- **`agent_id` resolution**: `--agent-id` (when non-blank) wins; else `TG_LEDGER_AGENT_ID`; else `TG_EVIDENCE_AGENT_ID` (shared fallback with `tg evidence`); else the literal string `"anonymous"`. Recorded verbatim, never inferred from process/user identity. Do not put secrets in `--agent-id` or `--note` -- both are written to a plaintext, per-repo, multi-agent-readable JSON file with no access control beyond the filesystem.
- **TTL and expiry**: default `900` seconds, overridable per-claim with `--ttl` or globally via `TG_LEDGER_CLAIM_TTL_SECONDS`. A dead agent's claim needs no special crash handling -- it simply stops appearing in `list`/overlap results once `expires_at` passes; physical removal from disk happens lazily on the next `claim`/`release` write (a pure `list` never rewrites the index). A record with a missing or unparseable `expires_at` (corrupt write, foreign edit) is treated as already-expired, not as immortal.
- **Concurrency and DoS bounds**: every write (`claim`/`release`) is a read-modify-write serialized under the same per-root `index_lock` primitive `tg session`/`tg checkpoint` use (cross-process safe, self-healing stale-lock reclaim, no lost inserts under concurrent writers). The on-disk index is capped at 256 live claims per root (oldest `created_at` evicted first) as a bound distinct from TTL expiry -- a flood of claims with a long TTL cannot grow the index without limit even though none of them are individually expired yet. The index read is byte-bounded before parsing (8 MiB) as a pre-parse DoS guard.
- **Not yet built (future slices, do not assume present)**: no MCP tool surface (CLI-only in Slice 1), no task-level or cross-repo coordination, no automatic claim renewal/heartbeat, no enforcement mechanism of any kind -- `tg ledger` reports; it never blocks, queues, or serializes an actual edit.

## 10. Ledger findings (`tg ledger record` / `find`) -- content-addressed reuse of computed artifacts (Slice 2)

`tg ledger record`/`find` extend the Slice 1 coordination plane (section 9) with a content-addressed artifact-reuse cache: one agent `record`s an evidence-receipt/blast-radius/context-pack/repo-map artifact it already computed; a sibling agent `find`s it by symbol and reuses the artifact instead of recomputing -- the "saves on searches / uses contracts" pillar. State lives at `<root>/.tensor-grep/ledger/findings/index.json` (a separate index from claims' `claims/index.json`) plus `<root>/.tensor-grep/ledger/findings/blobs/<receipt_sha256>.json` (one file per DISTINCT artifact, content-addressed and dedup'd -- recording the identical artifact twice writes the same blob path). Created only by the first `record` in a repo (default-inert: `find` never creates it, mirroring claims' `list`).

**Status: EXPERIMENTAL / preview (Slice 2)**, same posture as section 9 -- CLI surface and JSON schema may change in a minor release. `tg ledger record --help` / `tg ledger find --help` are marked `EXPERIMENTAL (Slice 2)`. **Default-OFF in the sense that matters**: nothing in `tg agent`/`tg edit-plan` consults the ledger automatically in this slice -- `record`/`find` are explicit-invoke only (auto-consult is a future, CEO-gated slice).

- **Exit-code contract for `record` (two states, same shape as section 9's claim/release/list)**: `0` = the artifact was ingested (or an identical artifact was already present -- dedup is not an error) and nothing else is asserted. `2` = a fail-closed condition fired and **nothing was written**: missing `--receipt`, an `--artifact-kind` outside `{blast-radius, evidence-receipt, context-pack, repo-map}`, a missing/oversized (>8 MiB)/non-JSON `--receipt` file, a findings-index lock-acquire timeout, or a write failure.
- **Exit-code contract for `find` (three states -- distinct from every other `tg ledger` subcommand, deliberately mirroring the `tg defs`/`callers`/`blast-radius` symbol-command contract in section 3)**: `0` = at least one returned finding is `fresh: true` (safe to reuse). `1` = zero returned findings are fresh (recompute) -- this covers both "no finding matched `--symbol`/`--artifact-kind` at all" and "matches exist but none are fresh," including when `--fresh-only` filtered every match away. `2` = a fail-closed condition fired: missing/blank `--symbol`, a corrupt/oversized findings index, or -- the case unique to `find` -- a blob that fails its integrity check (see below). A harness branches on this directly: `if tg ledger find PATH --symbol S --fresh-only; then reuse the artifact; else compute; fi`.
- **Finding record shape**: `{ledger_schema_version, kind: "finding", finding_id, agent_id, artifact_kind, symbol, receipt_sha256, blob_relpath, signed, key_id, revision, created_at, expires_at, ttl_seconds}`. `finding_id` is `finding-<UTC-compact-timestamp>-<8-hex>`. `receipt_sha256` is `tensor_grep.cli.evidence_signing.receipt_digest()` computed over the artifact JSON -- the SAME canonical-JSON content-address function `tg evidence` uses (`tg-canonical-json-v1`: compact, key-sorted, ASCII-only, `signature`/`receipt_sha256` fields excluded), so a plain (non-receipt) artifact hashes identically regardless of incidental whitespace, and an artifact that already happens to carry its own `receipt_sha256`/`signature` fields (a real EvidenceReceipt) is not self-referentially corrupted by them. `revision` is the same `_repo_revision_identity` shape sections 8/9 use. `signed`/`key_id` are captured at record time from the artifact's own (self-claimed, unverified) `signing` block presence -- never proof of authenticity by themselves (see the trust-bootstrap rule below).
- **`tg ledger record PATH --receipt FILE.json [--artifact-kind K] [--symbol S] [--agent-id ID] [--ttl SECONDS] [--json]`**: `--artifact-kind` defaults to `evidence-receipt` and is validated against the fixed enum at WRITE time (rejects garbage before it enters the store). `--receipt` is read bounded (refuses over 8 MiB before parsing -- the same "pre-auth unbounded read" DoS guard section 8/9 use elsewhere) and must parse as a JSON object. Response JSON: `{version, schema_version, routing_backend: "Ledger", routing_reason: "ledger-record", sidecar_used: false, ledger_schema_version, advisory: true, finding: {...}}`.
- **`tg ledger find PATH --symbol S [--artifact-kind K] [--fresh-only] [--trusted-key KEY ...] [--json]`**: `--symbol` is required (validated in the store, not by Typer, so a missing value still returns the standard `{"error": {...}}` JSON envelope under `--json` rather than a bare Click usage error). `--artifact-kind`, UNLIKE `record`, is a plain string-equality filter at READ time -- not validated against the enum, so an unrecognized value simply matches nothing rather than erroring. Response JSON: `{..., routing_reason: "ledger-find", findings: [...], count, any_fresh}`; each entry in `findings` is the finding record plus `fresh: bool` and `key_trusted: bool | None` (`null` when the finding is unsigned or no `--trusted-key`/`TG_EVIDENCE_TRUSTED_KEYS` was supplied).
- **Freshness (revision-primary, never silently served as current)**: a finding is `fresh: true` only when BOTH its captured `revision` and the CURRENT repo's `_repo_revision_identity(root)` resolved to `status: "present"` AND their `commit_sha` AND `dirty_tree_sha256` both match exactly -- so a single uncommitted edit anywhere in the tracked/untracked diff (which changes `dirty_tree_sha256`) flips a finding to `fresh: false` even on the same commit. An unresolvable revision on EITHER side (not a git repo, git unavailable) fails closed to `fresh: false`, never fabricated. Both capture (at `record` time) and the current-state recheck (at `find` time) exclude the ledger's own `.tensor-grep/ledger/` subtree from the dirty-tree computation (`_repo_revision_identity(..., exclude_prefixes=(".tensor-grep/ledger",))`, mirroring section 8's `tg codemap --check` self-dirty fix) -- otherwise the very act of `record`ing would make the repo read as dirty against itself and poison every subsequent freshness check. This is a wall-clock-independent check computed fresh on every `find` call; the TTL below is a separate, secondary backstop.
- **Integrity (fail-closed, never serve tampered data)**: every finding actually being returned (i.e. surviving the `--symbol`/`--artifact-kind`/`--fresh-only` filters) has its blob read back and its content digest recomputed via `receipt_digest`; a mismatch (or a missing/oversized/non-JSON blob) raises and the WHOLE `find` call exits `2` -- it never silently drops the bad finding and returns the rest, and never serves unverified content. A finding excluded by `--fresh-only` is never integrity-checked (only findings about to be served are). `--trusted-key`/`TG_EVIDENCE_TRUSTED_KEYS` additionally re-verifies a `signed` finding's embedded signature against the CALLER's current trust config (which may differ from whatever was configured at record time) to compute `key_trusted`; this mirrors section 8's trust-bootstrap rule exactly -- an embedded public key alone proves internal consistency, never authenticity.
- **TTL, cap, and GC**: a wall-clock backstop TTL defaults to `86400` seconds (24h; overridable per-record with `--ttl` or globally via `TG_LEDGER_FINDING_TTL_SECONDS`) -- deliberately far longer-lived than a claim's 900s default, since a finding is a reusable artifact pointer, not a short-lived coordination signal. The index is capped at 512 live findings per root; total on-disk bytes across all DISTINCT (dedup'd) blobs is separately capped (default 256 MiB, `TG_LEDGER_MAX_BLOB_BYTES`) since a flood of small findings could stay under the count cap while a few large artifacts still grow disk unboundedly. Either cap evicts the oldest `created_at` findings first. Every `record` call prunes TTL-expired findings and garbage-collects any blob no longer referenced by a live finding (a shared/dedup'd blob is kept until its LAST referencing finding is gone).
- **Not yet built (future slices, do not assume present)**: no automatic `tg agent`/`edit-plan` consultation (explicit-invoke only in Slice 2), no MCP tool surface, no manual `release`/delete for a findings entry (rely on TTL/cap eviction), no cross-repo lookup.
